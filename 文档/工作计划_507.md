## 2. DemoSpeedup 官方实现要点

官方 README 中给出的主流程是：

1. 先在原始 demonstrations 上训练 proxy policy。
2. 使用 proxy policy 标注 demo entropy。
3. 使用带 entropy label 的 accelerated demos 训练最终策略。

Aloha 入口体现为：

- `--label`：用 proxy policy 生成 entropy / label。
- `--speedup`：训练时读取 label，并对 action chunk 做加速。
- 官方建议加速策略训练时把 chunk size 减半，以保持 geometrical consistency。

Bigym / robobase 入口体现为：

- `label.py` 读取 `best_snapshot.pt`，调用 `workspace.label()`。
- `workspace.label()` 多次采样 action，调用 KDE entropy，随后用 HDBSCAN 生成 labels。
- replay buffer 在 `speedup=True` 时读取 labels，并用 label-guided downsample 改写 action sequence。

关键实现细节：

- entropy 估计：`KDE.kde_entropy(...)` 对多次 action samples 做 Gaussian kernel density 估计。
- 分段：先用 IsolationForest 替换异常 entropy，再把归一化时间 index 和归一化 entropy 拼成二维点，使用 HDBSCAN 聚类。
- label 语义：`0` 更接近 precision / low entropy，`1` 更接近 casualness / high entropy。
- action 下采样：官方默认 `low_v=2`、`high_v=4`；当连续 `label==1` 时用更大的步长。

对当前 seatbelt 项目的启发：

- 1500 step LoRA baseline 可作为当前最稳的 proxy policy。
- 不需要重写视频或完整 parquet；可以只生成 action index map，在 dataset 取 action chunk 时按 map 读取未来动作。
- 第一版应比官方 `{2, 4}` 更保守，先用 `{1, 2}` 或 `{1, 3}`，避免破坏安全带插入、脱离、碰撞恢复等精细动作。

## 3. 当前适配原则

### 3.1 先评测 1500 baseline

在做 DemoSpeedup 前，必须先评测 1500 step baseline。原因：

- 它将作为 DemoSpeedup 的 proxy policy。
- 它也是后续 speedup LoRA 的更强对照线。
- 如果 1500 baseline 已经显著改善 `take_off_move`，则 DemoSpeedup 的收益判断要以 1500 而不是 500 step baseline 为准。

评测沿用 `baseline实验协议_506.md`：

- 8 个固定 test repo。
- `sample_steps=4`
- `num_batches=10`
- `duration_dims=0-5,7-12`
- 输出 `duration_summary.json`、`duration_per_chunk.csv`、`test_all_preds.npy`、`test_all_gts.npy` 和可视化图。

1500 baseline 结果输出建议目录：

- `eval_results/baseline_1500_507/test_results`
- `eval_results/baseline_1500_507/vis`

### 3.2 DemoSpeedup 第一版只做局部保守加速

第一版目标任务：

- P0：训练集中的 `take_off_move`
- P1：如果 P0 有收益，再扩展到 `insert_move`
- P2：最后再考虑全量 15 个 repo

原因：

- 500 step baseline 中 `take_off_move` 是最慢 seen task，speedup factor 只有 `0.540`。
- `take_off_move` 在训练集中有 3 个 repo，可做数据侧加速。
- 其他任务如 `avoid_obstacle`、`hang` 已接近持平或略快，过早全量加速可能引入退化。

### 3.3 不直接复制官方速度比例

官方实现中常用：

- precision / low entropy：`2x`
- casualness / high entropy：`4x`

当前 seatbelt v0 建议：

| 版本 | precision 段 | casualness 段 | 用途 |
| --- | ---: | ---: | --- |
| `ds_takeoff_v0` | `1x` | `2x` | 最小风险，验证管线 |
| `ds_takeoff_v1` | `1x` | `3x` | v0 稳定后增强 |
| `ds_takeoff_v2` | `2x` | `4x` | 接近官方实现，放到后续 |

### 阶段 B：实现 entropy 标注工具

输入：

每帧记录：

```json
{
  "repo_id": "...",
  "episode_index": 0,
  "frame_index": 123,
  "timestamp": 4.1,
  "entropy_kde": 0.0,
  "action_std_mean": 0.0,
  "num_action_samples": 8
}
```

实现细节：

- 对同一个 observation 用不同 RNG 多次调用 `sample_actions`。
- 第一版可同时保存两种分数：
  - `action_std_mean`：简单、稳定、容易 debug。
  - `entropy_kde`：参考官方 KDE 实现。
- 如果 KDE 计算太慢，先用 `action_std_mean` 跑通 v0。

### 阶段 C：实现 entropy -> label

新增脚本建议：

`openpi_modified/scripts/demospeedup/build_entropy_labels.py`

输入：

- 阶段 B 输出的 entropy JSONL。

输出：

- `eval_results/demospeedup_507/labels/<repo_id>/episode_*.npy`
- `eval_results/demospeedup_507/labels/<repo_id>/episode_*_label.png`
- `eval_results/demospeedup_507/labels/label_summary.json`

v0 分段策略：

- 不强依赖 HDBSCAN。
- 对每个 episode 做滑窗平滑。
- episode 内分位数分段：
  - entropy 高于 `q60`：`label=1`，casualness。
  - entropy 低于 `q40`：`label=0`，precision。
  - 中间区域根据相邻 label 或滑窗均值归类。

v1 分段策略：

- 加入 `hdbscan` 依赖。
- 复刻官方思路：
  - IsolationForest 处理 outlier。
  - `[time_index_norm, entropy_norm]` 二维聚类。
  - 低 entropy cluster 标为 precision。
  - 其他点标为 casualness。

注意：

当前 `openpi_modified` 没有看到 `hdbscan` 依赖，因此 HDBSCAN 不作为 v0 blocker。

### 阶段 D：实现 label-guided action index map

新增脚本建议：

`openpi_modified/scripts/demospeedup/build_action_index_map.py`

输入：

- labels
- 原始 repo parquet metadata
- `action_horizon=50`
- `r_precision=1`
- `r_casual=2`

输出：

- `eval_results/demospeedup_507/index_maps/ds_takeoff_v0/<repo_id>/episode_*.npz`
- `eval_results/demospeedup_507/index_maps/ds_takeoff_v0/summary.json`

index map 格式：

```text
map[frame_index, horizon_offset] -> target_frame_index
```

规则：

- observation 仍然来自原始 `frame_index=t`。
- action chunk 不再简单取 `t, t+1, ..., t+49`。
- action chunk 改为按 label-guided speedup indices 读取。
- 超出 episode 末尾时沿用当前 dataset 的 padding / clamp 逻辑。

v0 下采样逻辑：

```text
if label == 1:
    step += 2
else:
    step += 1
```

官方兼容版本：

```text
if label == 1 and next high_v labels are all 1:
    step += high_v
else:
    step += low_v
```

## 6. 风险与止损

### 风险 1：entropy estimator 不可靠

表现：

- entropy 曲线无明显阶段性。
- 高 entropy 出现在精细接触段。

止损：

- 降低对 entropy 的依赖，改为 `action_std_mean + action_delta + gripper/state heuristic` 混合分数。
- 只加速 entropy 最高的 top 25% 帧。

### 风险 2：加速后 MSE 明显恶化

表现：

- `take_off_move` speedup 提升，但 MSE / MAE 大幅变差。

止损：

- 从 `{1, 2}` 退到只加速连续高 entropy 长段。
- 保留原始数据和加速数据混训，例如同一 repo 中 50% 原始 chunk、50% speedup chunk。

### 风险 3：dataset index map 破坏 episode 边界

表现：

- action chunk 跨 episode。
- padding / mask 与原始数据不一致。

止损：

- index map 只在 episode 内生成。
- map 构建阶段强制 assert 所有 target index 在 `[episode_start, episode_end)`。
- dataset smoke test 先覆盖 episode 头尾各 20 帧。

### 风险 4：open-loop proxy 提升不代表闭环成功

处理：

- 报告中继续明确命名为 `open-loop duration speedup factor`。
- 不把该指标等同于真实任务完成时间。
- 一旦 open-loop 有稳定收益，下一步接 lerobot 部署日志或小规模真机闭环。
