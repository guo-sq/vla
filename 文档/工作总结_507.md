## 2. DemoSpeedup 方向规划

本轮发现并参考了 DemoSpeedup 官方实现：

`https://github.com/lingxiao-guo/DemoSpeedup`

论文与官方实现的核心流程是：

1. 先在原始 demonstrations 上训练 proxy policy。
2. 用 proxy policy 多次采样动作，估计 action entropy。
3. 根据 entropy 曲线划分 precision / casualness 阶段。
4. 对不同阶段使用不同加速倍率，生成 accelerated demonstrations。
5. 再用加速后的 demonstrations 训练最终 policy。

结合当前 `openpi_modified`，本轮确定不直接迁移官方 ACT / DP / robobase 框架，而是适配三块逻辑：

- proxy policy entropy / uncertainty 标注。
- entropy 到阶段 label 的转换。
- label-guided action index map 或数据重写。

第一版建议保持保守：

| 版本 | precision 段 | casualness 段 | 目的 |
| --- | ---: | ---: | --- |
| `ds_takeoff_v0` | `1x` | `2x` | 最小风险验证管线 |
| `ds_takeoff_v1` | `1x` | `3x` | v0 稳定后增强 |
| `ds_takeoff_v2` | `2x` | `4x` | 接近官方倍率，后续再试 |

当前不建议马上全量加速 15 个训练 repo。优先从 `take_off_move` 这类诊断价值高的任务开始。

## 5. seen / unseen 含义重新明确

当前测试不是严格意义上的“泛化到所有真实场景”。

`test_seen_task`：

- 任务类型在训练中见过。
- 但测试 repo / batch 未放入 LoRA 训练集。

- 更接近 task-level generalization。
- 但基础 seatbelt checkpoint 可能已经见过相关 seatbelt 分布，因此不能等同于完全从零泛化。

后续报告应把 `test_seen_task` / `test_unseen_task` 表述为 LoRA 层面的 seen / unseen，而不是整个系统从未见过。

## 7. 新增 GT-target completion 指标

旧 speedup 指标的问题：

- Pred 和 GT 分别看自己什么时候接近自己的最终动作。
- 如果 Pred 很快收敛到错误终点，也会被判定为“很快完成”。
- 因此旧指标会把“早塌缩”误判为“动作加速”。

本轮新增离线复评脚本：

`/root/workspaces/wujie_gsq/vla/openpi_modified/scripts/analyze_gt_target_completion.py`

新指标核心：

`pred_to_gt_final_completion_time`

它计算：

- GT 什么时候稳定到达 GT chunk 的最终动作。
- Pred 什么时候稳定到达同一个 GT chunk 的最终动作。
- speedup 仍用 `gt_time / pred_time`，但 Pred 的目标不再是自己的终点，而是 GT 终点。

容差定义：

`target_tolerance = max(abs_tol, rel_tol * GT selected-dim path length)`

这样可以避免 Pred 自己抖动或路径长度异常导致容差被放大。

输出文件：

- `gt_target_completion_summary.json`
- `gt_target_completion_per_chunk.csv`
- `gt_target_completion_aggregate.csv`
- `gt_target_completion_aggregate_summary.json`

## 9. 当前总体判断

截至 5/07，当前项目训练部分还没有真正做动作加速改造。

已经完成的是：

- LoRA baseline 训练。
- 500 / 1500 step baseline 对照。
- open-loop duration proxy 指标。
- action-space 轨迹可视化。

尚未完成的是：

- DemoSpeedup entropy 标注。
- entropy-guided 数据重写。
- speed / quality / mistake metadata conditioning。
- 真正 CFG 推理。
- RTC 消融训练。
- 闭环或真机完成时间评估。

因此不建议现在继续盲目重训更长 step。当前最应该先做的是：

1. 把新指标纳入固定实验协议。
2. 用新指标更新后续所有 baseline / speedup 报告。
3. 再开始 DemoSpeedup v0 的数据侧加速实验。
