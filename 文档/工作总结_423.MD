# 工作总结 4/23

## 1. 当前项目总链路

当前项目可以概括成这条主链路：

`原始数据 -> 优化数据 -> 训练模型 -> 应用至机器人`

但结合实际代码，更准确的版本是：

`原始/已有数据 -> 数据筛选/重加权/快轨迹接入 -> 模型训练(RTC/optimality) -> 部署运行时加速(async/chunk/fusion) -> 机器人执行`

结论：

- 这条链路是对的。
- 但“优化”不只发生在数据侧，也发生在训练侧和部署侧。

## 2. 各环节当前已经完成的工作

### 2.1 原始数据 -> 优化数据

这一段目前是“部分完成”。

已经完成的工作：

- `openpi_modified` 已经有数据预处理链，而不是直接拿原始 episode 生训。
- 已有 `valid_mask`、`sample_weight`、`static_detector` 这套机制，负责去掉静止段、无效段、降低低价值帧采样权重。
- 已有 `optimality_preprocessor`，会给帧打 `optimality` 标签。
- 已有 `use_state_as_action` 开关，为“轨迹重定义/加速监督目标”留出了通路。
- 项目里已经接入了一批 `fast` 原始数据，不是完全只有慢轨迹。

和 paper 的对应关系：

- 和 `DemoSpeedup` 的核心思想接近。
- 但当前还没有看到一条完整的：
  `原始慢轨迹 -> 自动分段 -> 自动加速 -> 生成新数据集`
  自动流水线。

结论：

- 有“筛选”和“重权重”。
- 有“快轨迹直接接入训练”。
- 没有完整自动 speedup dataset 生成器。

### 2.2 优化数据 -> 训练模型

这一段是当前项目里做得最完整的一段。

已经完成的工作：

- RTC 训练机制已经实装在 `openpi_modified/src/openpi/models/pi0.py`。
- `rtc_max_delay` 已经进入训练入口 `openpi_modified/scripts/train.py`。
- 推理时 `infer_delay + action_prefix` 已打通到 `openpi_modified/src/openpi/policies/policy.py`。
- `optimality` 条件化已经进入模型输入。
- 多个 config 已经在使用 `rtc_max_delay`。

和 paper 的对应关系：

- 和 `Training-Time Action Conditioning for Efficient Real-Time Chunking` 非常接近，可以认为核心训练机制已在仓库里。
- 和 `DemoSpeedup` 的“质量感知训练”也有相似性，但当前更像条件化训练，不是自动熵引导加速。

结论：

- 代码不是空白。
- 当前主要缺的是实验验证和消融结果，而不是实现本身。

### 2.3 训练模型 -> 应用至机器人

这一段也已经有不少工作。

已经完成的工作：

- `lerobot_modified` 有 `ActionBuffer`、`compute_optimal_infer_interval()`、`inference_speedup`。
- 已有 chunk fusion、`fusion_window`、`smooth_sigma`。

和 paper 的对应关系：

- 和 `SAIL`、`VLASH`、`Realtime-VLA V2` 的系统侧问题高度接近。

结论：

- 部署侧基础设施已经很强。
- 当前主要缺的是系统实验和真机闭环验证。
