# 基础模型与数据划分问题分析报告 508

本文整理当前 seatbelt VLA / LoRA baseline 实验的训练集、测试集、评测协议和几次训练模型质量。重点不是只判断“哪个 checkpoint 分数高”，而是回答一个更根本的问题：现在看起来模型效果不好，究竟更像是训练集问题、测试集问题、评测指标问题，还是训练目标本身没有对齐“动作更快”。

## 1. 当前训练到底训练了什么

当前几次实验不是从零训练一个 seatbelt 基础模型，而是在已有 pi0.5 seatbelt base checkpoint 上做 LoRA 微调。

base checkpoint：

```text
/mnt/oss_models/models_deploy/2603/seatbelt/
pi05_base_seatbelt_data_0205_0312_horizon_50_trtc_single_self_play_recovery/
pi05_base_seatbelt_data_0205_0312_horizon_50_trtc_single_self_play_recovery_exp/84999
```

LoRA baseline 配置来源：

```text
vla/openpi_modified/src/openpi/configs/cfg_baseline_seatbelt_lora_1500_507.py
```

模型相关设置：

| 项 | 设置 |
| --- | --- |
| base model | pi0.5 seatbelt base checkpoint `84999` |
| Paligemma variant | `gemma_2b_lora` |
| action expert variant | `gemma_300m_lora` |
| `enable_recap` | `False` |
| `enable_rl_value_head` | `False` |
| `ema_decay` | `None` |

注意：当前 LoRA baseline 的训练目标仍然是 imitation / action prediction，没有显式加入“更快完成动作”的 supervision，也没有 DemoSpeedup 的加速标签、entropy guide、index map 或 speed prompt。因此，如果只靠 LoRA 多训几步，理论上更可能降低 MSE，而不是自然学出显著快于示教的动作。

从统计看，训练集有几个明显特点：

## 4. 评测协议

所有正式评测保持固定协议：

| 参数 | 值 |
| --- | --- |
| `num_batches` | `10` |
| `batch_size` | `1` |
| `sample_steps` | `4` |
| `num_workers` | `0` |
| `duration_analysis` | `1` |
| `duration_dims` | `0-5,7-12` |
| `duration_abs_tol` | `0.02` |
| `duration_rel_tol` | `0.05` |
| `duration_stable_steps` | `3` |
| `lazy_load_eval` | `1` |

duration dims 使用 `0-5,7-12`，跳过了 gripper-like 维度 `6,13`。action-space 轨迹图默认把 `0,1,2` 当左臂 XYZ-like 轨迹，把 `7,8,9` 当右臂 XYZ-like 轨迹。它不是机器人真实末端位姿，只是 action-space 直观对比。

目前指标有两类：

1. 旧 speedup：比较 pred 自己的稳定/完成时间与 GT 的稳定/完成时间，容易在“轨迹一开始就接近终点”或“预测几乎不动”时虚高。
2. target-completion speedup：看 pred 何时到达 GT final target，更接近“是否真的完成到示教终点”。本报告优先参考这个指标和 `pred_reached_gt_final_rate`。

直观结论：

1. `bs8-1000` 的 MSE/MAE 最低，说明它是当前最像 GT 的拟合型 checkpoint。

5. old speedup 在 `bs1-1500` 和 `bs8-1000` seen 上明显虚高，主要被 `take_off_move` 拉爆，不可信。

## 7. 关键问题 1：old speedup 有明显假快

`take_off_move` 的旧指标与 target-completion 指标对比：

![takeoff_old_vs_target](analysis_report_assets_508/takeoff_old_vs_target.png)

`take_off_move` 分项：

| 实验 | MSE | old speedup | target speedup | reached | final err |
| --- | ---: | ---: | ---: | ---: | ---: |
| bs1-500 | 0.000141 | 0.540 | 1.255 | 80% | 0.107 |
| bs1-1500 | 0.000049 | 11.476 | 1.095 | 60% | 0.033 |
| bs8-500 | 0.000094 | 0.609 | 1.066 | 90% | 0.082 |
| bs8-1000 | 0.000008 | 24.100 | 0.945 | 50% | 0.029 |
| bs8-1500 | 0.004724 | 0.553 | 1.759 | 80% | 0.469 |

这说明旧 speedup 不能作为主优化目标：

- `bs8-1000` 的 old speedup 达到 `24.1`，但 target speedup 只有 `0.945`，并没有真正更快到达 GT final target。
- `bs8-1500` 的 target speedup 达到 `1.759`，但 final err 达到 `0.469`，说明可能是中途接近过目标但最终偏离明显。
- 低 MSE 不等于加速。`bs8-1000 take_off` MSE 极低，但 target speedup 低于 1。

## 8. 关键问题 2：seen 也不稳定，说明不只是泛化问题

`test_seen_task` 是同任务类型不同 repo/batch，按理应该比 unseen 容易。但结果显示：

- `hang` 一直是 seen 中最差的一项，MSE 高、final err 高、reached rate 经常为 0。
- `avoid_obstacle` MSE 低，但 target speedup 常年约 `0.992`，几乎没有加速。
- `insert_move` 比较稳定，是少数 target speedup > 1 且 reached 不太差的任务。
- `take_off_move` 指标高度不稳定，容易出现旧 speedup 虚高。

`bs8-1000 hang` 轨迹示意：

![traj_bs8_1000_hang](analysis_report_assets_508/traj_bs8_1000_hang.png)

这说明问题不只是“unseen 任务太难”。在 seen 任务里，模型也没有稳定学到“同任务新 repo 的快速完成策略”。当前 LoRA baseline 更像是在局部拟合 action 分布，而不是形成可靠任务策略。

## 10. 当前最可能的问题来源

我把可能性按优先级排序：

### 10.1 训练目标没有对齐“加速”

当前 LoRA 训练仍是普通 imitation。它优化的是动作接近 GT，而不是更早到达 GT final target。MSE 降低时，动作可能更像示教，也可能更慢、更保守。

证据：

- `bs8-1000` MSE/MAE 最好，但 seen reached 从 `bs8-500` 的 `57.5%` 降到 `25.0%`。
- `bs8-1000 take_off` MSE 极低，但 target speedup < 1。
- 继续增加 step 到 1500 没有稳定提升泛化。

### 10.2 评测指标仍有脆弱点

旧 speedup 会被 `take_off_move` 明显误导。target-completion 已经更好，但它仍然只是 action-space final target，不等于真实任务成功。

后续必须把主要判断从 old speedup 切到：

- target speedup；
- reached rate；
- final target distance；
- action-space 轨迹图；
- 最终最好接真实 rollout / task success。

### 10.3 测试集划分有“过难”和“样本少”两类问题

`test_seen_task` 合理，但每个 repo 只评测 10 个 batch，统计方差偏大。`test_unseen_task` 则有两类问题：

- 任务类型完全没进 LoRA 训练集，如 `collision_recovery`、`back_home`。
- `fix_insert_move` 只有 4 episodes，作为泛化结论依据太弱。

因此，当前 unseen 更适合作为 stress test，不适合作为唯一模型选择标准。

### 10.4 训练集覆盖和均衡不足

训练集只有 15 个 repo，覆盖 4 类任务。对于 seatbelt 全任务空间来说偏窄，而且各任务帧数差异明显。尤其是希望模型具备更快、更稳、更泛化的能力时，只靠这些 repo 做 LoRA 不够。

此外，当前没有显式说明 repo sampling 是严格均匀还是按帧数/样本数加权。如果按样本数加权，长轨迹任务如 `take_off`、`hang` 会占更多采样概率；如果按 repo 均匀，又可能每个 repo 内短/长 episode 贡献不均。这一点建议后续查代码或直接记录 dataloader 采样分布。

### 10.5 base checkpoint 的“真实训练覆盖”尚未核实

当前 seen/unseen 是相对于 LoRA 训练集定义的。但 base checkpoint 可能已经见过更多 `0205_0312` seatbelt 数据。如果要判断“基础模型本身是否有问题”，应该单独评测未 LoRA 的 base checkpoint `84999`，并核对 base 训练 repo 是否包含当前 test repo。

## 11. 我的判断

现在不能简单说“基础模型完全坏了”，但可以说：

1. 当前 LoRA baseline 没有稳定学出“显著快于示教且到位”的动作。
2. 单纯增加 step 或 batch size 不是主解法；它主要改变拟合程度，不会自动产生加速策略。
3. 训练集本身覆盖偏窄，对 unseen 任务泛化要求过高。
4. 旧 speedup 指标已经被证明会产生严重假快，不能再作为主指标。
5. `bs8-500` 更像当前“到达率较好”的候选，`bs8-1000` 更像“拟合误差较低”的候选，二者都不是合格的 speedup 模型。

## 12. 建议下一步

我建议按下面顺序排查，而不是继续盲目训练：

1. 单独评测 base checkpoint `84999`，不加 LoRA，使用同一 seen/unseen 协议。这样能判断 LoRA 是否在破坏 base，还是 base 本来就不稳。
2. 把 `test_seen_task` 扩到每 repo 至少 50 或 100 chunks，降低 10 batch 带来的随机性。
3. 给测试集增加一个“train-repo holdout chunk”切分：同 repo 不同 episode/chunk，判断模型是否连训练分布内都不稳。
4. 对训练集做数据质量扫描：静止比例、动作幅度、episode 长度、final target 分布、失败 episode、速度分布。
5. 进入 DemoSpeedup 路线：先基于 entropy / target completion 生成保守加速 index map，不要再期待普通 LoRA 自动加速。
6. 把优化目标从 old speedup 换成组合指标：`target_speedup > 1`、`reached_rate` 不下降、`final_err` 不恶化、MSE/MAE 不爆。
