# 工作计划 5/06

## 1. 当前任务目标

当前阶段的核心目标是：

`在当前 pi0.5 / seatbelt 项目基础上，实现并验证动作加速能力。`

这里的“动作加速”优先指机器人动作序列本身更快完成任务，而不是仅仅让模型推理更快。

阶段目标可以拆成三步：

1. 已在 L20 上用 LoRA 打通小规模训练与测试，并完成 Seatbelt Baseline v1。
2. 下一步引入 speed-conditioned 训练，让模型能根据速度条件输出更快动作。
3. 在 baseline v1 的固定训练/测试协议上做横向对比，再决定是否上 DLC 做正式训练。

### 2.3 已有评测指标

当前 `test.py` 已支持：

- MSE
- MAE
- 模型输出动作估计耗时
- 示教动作估计耗时
- speedup factor
- faster / same / slower chunk rate
- per-chunk CSV 明细

需要特别注意：

- 当前 `speedup_factor_gt_over_pred` 是 open-loop chunk 内部的动作收敛速度估计，不是真机闭环任务完成速度，也不是模型推理速度。
- 计算方式是 `gt_mean_seconds / pred_mean_seconds`。
- `pred_mean_seconds` / `gt_mean_seconds` 来自每个固定长度 action chunk 内“从第几帧开始连续若干帧接近该 chunk 自己的末端动作”。
- 因此 `speedup_factor > 1` 只表示模型预测动作 chunk 比 GT chunk 更早收敛到自身末端动作；它不直接保证 pred 到达 GT 目标、不保证成功率，也不代表推理延迟降低。
- 后续报告中应把这个指标明确命名为 `open-loop duration speedup factor`，避免和真实执行加速或 denoising / inference speedup 混淆。

下一阶段所有实验都应至少输出：

- `duration_summary.json`
- `duration_per_chunk.csv`
- `test_all_preds.npy`
- `test_all_gts.npy`
- 轨迹可视化图

### 阶段 B：两条 P0 主线并行

目标：

同时推进数据侧动作加速和条件化训练侧动作加速，先拿到最小可验证结果。

需要先拆清两类“加速”：

| 类型 | 目标 | 当前适合方法 |
| --- | --- | --- |
| 动作执行加速 | 让模型输出本身更快完成任务的动作轨迹 | DemoSpeedup、pi0.7 metadata / speed conditioning、后续 VLA-RL |
| 推理延迟加速 | 让同样或近似轨迹更快生成出来 | SnapFlow、FASTER、PD-VLA、部署侧异步调度 |

当前 baseline v1 的 `speedup_factor` 只属于第一类的 open-loop proxy，不能和 SnapFlow / FASTER 的 denoising 或 reaction latency 数字混用。

#### B.1：pi0.7-lite 升级版 metadata conditioning

计划：

- 保留原计划中的 episode speed 统计，基于有效动作时长生成 `fast / normal / slow` 标签。
- 不只注入 speed 单项，至少在 prompt / metadata 中加入三类条件：
  - `Quality=5`
  - `Speed=fast`
  - `Mistake=false`
- 训练时对这些 metadata 做随机 dropout，先模拟 pi0.7 的 classifier-free guidance 训练分布。
- 推理时固定使用 `Quality=5, Speed=fast, Mistake=false`。
- 第一版先做 prompt / transform 层注入；真正的 CFG 推理需要 conditional / unconditional 两路 action prediction，可放到 P1/P2。
- `Mistake` 标签优先从已有 `is_human_intervention`、self-play failure / recovery 字段自动生成。
- `Quality` 标签第一版可先用 episode 是否成功、是否 intervention、是否 recovery、动作平滑度等规则近似打分。

需要实现的位置：

- 数据预处理或 transform 层：
  - `openpi_modified/src/openpi/transforms.py`
  - 或新增 dataset/frame attribute preprocessor
- 配置层：
  - 新增一个 seatbelt speed LoRA config
- 统计输出：
  - `eval_results/speed_label_506/episode_speed_stats.json`
  - 或等价 CSV/JSON 汇总

产出：

- episode speed / quality / mistake 统计文件。
- metadata-conditioned 数据加载链路。
- 新的训练配置文件。

#### B.2：DemoSpeedup 最小可行版

目标：

把 DemoSpeedup 从“占位想法”提前为 P0 小实验。先不全量改数据，只针对 baseline v1 中最慢、最有诊断价值的任务类型做保守加速。

计划：

- 使用 baseline v1 LoRA checkpoint 作为第一版 entropy / uncertainty estimator。
- 对 15 个训练 repo 计算 frame-level 或 segment-level 可加速分数。
- 第一版优先只处理 `take_off_move`，并在有训练数据时扩展到 recovery / collision 相关片段。
- 下采样策略保持保守，例如只对高熵或明显自由移动段做 `1.2x~1.5x` 压缩，避免破坏高精度接触动作。
- `optimality_preprocessor.py` 和 frame attribute pipeline 可复用来承载 frame score / sample weight，但真正的 DemoSpeedup 需要额外的数据重写工具，保证 image、state、action、timestamp、episode index 对齐。
- 生成一版 speedup demonstrations，再用 baseline v1 同样 LoRA 设置训练。

为什么先局部做：

- baseline v1 的慢点集中在 `take_off_move` 和 `collision_recovery` 一类场景，局部修改更容易看出因果。
- 其他任务已经接近持平或略快，直接全量下采样可能破坏原本还可以的分布。
- 当前还没有真机闭环结果，保守局部加速比追求论文中的最大 3x 更稳。

产出：

- frame / segment entropy 或 uncertainty 分数。
- speedup data rewrite 脚本或预处理说明。
- 局部 speedup demonstrations。
- 与 baseline v1 同协议的 LoRA 训练和测试结果。

### 阶段 C：RTC 最小消融

目标：

验证项目里已经实现的 `rtc_max_delay` 对 seatbelt LoRA 是否有收益。这条线开发成本最低，应该作为 P0/P1 的高 ROI 实验。

计划：

- 基于 `cfg_baseline_seatbelt_lora_506.py` 派生三组配置。
- 除 `rtc_max_delay` 外，其他训练集、base checkpoint、LoRA 参数、训练步数、评测协议全部保持一致。
- 先跑：
  - `rtc_max_delay=0`
  - `rtc_max_delay=5`
  - `rtc_max_delay=15`
- open-loop 阶段除了 MSE / MAE / duration proxy，还应补 chunk continuity 指标：
  - 相邻 chunk 首尾 action jump
  - overlap 区域差异
  - delay prefix 后的恢复误差
- 如果能接入部署日志，再比较固定 infer delay 下的 queue underrun、tracking error、动作突变。

对照组：

| 组别 | 说明 |
| --- | --- |
| GT | 测试集示教动作 |
| LoRA baseline v1 | 不带速度条件，已完成 |
| RTC LoRA | 仅修改 `rtc_max_delay` |
| Metadata-conditioned LoRA | 推理时指定 `Quality=5, Speed=fast, Mistake=false` |
| DemoSpeedup-data LoRA | 使用局部加速后的 demonstrations |
| speedup 历史模型 | 作为历史参考，待代码兼容后评测 |

评测指标：

- MSE / MAE 不应显著恶化。
- open-loop duration proxy 应优于 baseline，或至少不恶化。
- faster chunk rate 应提升。
- chunk continuity / action jump 不应恶化。

### 阶段 D：组合训练与严格 CFG

目标：

在单项实验有效后，再组合 DemoSpeedup 数据和 metadata conditioning，并实现更接近 pi0.7 的 CFG 推理。

计划：

- 组合 `DemoSpeedup-data + Quality/Speed/Mistake metadata`。
- 在 sampling 路径中实现 conditional / unconditional 两路预测，再按 guidance scale 组合。
- 对 guidance scale 做小范围 sweep，避免过强引导导致动作过激或偏离 GT。
- 如 `subtask_index` 覆盖较好，再尝试 subtask-aware DemoSpeedup：不同子阶段使用不同最大下采样率。

产出：

- 组合训练配置。
- CFG 推理配置和 guidance scale sweep。
- 与 baseline v1、RTC、单独 metadata、单独 DemoSpeedup 的横向对比。

风险：

- CFG 会增加推理开销，必须和推理延迟优化线分开汇报。
- metadata 自动标签可能有噪声，需要保留 ablation。
- subtask-aware / ESPADA-style 需要更可靠的语义分段，不建议第一步就做全量复杂版本。

### 阶段 E：RECAP / value / advantage 增强

目标：

如果 metadata + DemoSpeedup 对 open-loop proxy 有效果，再引入 pi0.6 / RECAP 的 value / advantage 机制，提升动作质量和速度稳定性。

计划：

- 启用 `enable_rl_value_head=True`。
- 使用 `ValueReturnsPreprocessor` 构造进度或剩余时间监督。
- 用 `AddAdvantageToPrompt` 加入 advantage 条件。
- 优先用已有 self-play / recovery 数据判断哪些片段是 positive / negative。
- 将 RECAP 结果和 metadata / DemoSpeedup 分开消融，不要直接混成一个大实验。

产出：

- value head 训练配置。
- RECAP/advantage conditioning 对动作速度、动作质量和成功趋势的影响对比。

风险：

- value 训练比 LoRA prompt 条件更复杂。
- 需要更可靠的成功/失败/完成时间标签。
- 可能更适合放到 DLC 上跑。

### 阶段 F：推理延迟与部署侧优化

目标：

在动作执行加速主线稳定后，再单独推进 chunk 生成延迟和异步部署体验。

计划：

- FASTER / Horizon-Aware Schedule 作为优先候选，因为它偏 plug-and-play，且和 flow-based VLA 匹配。
- SnapFlow 作为 pi0.5 专项 self-distillation 候选，收益大但工程改动更重。
- PD-VLA、Legato / A2C2、VLASH / VLA-RAIL 放入部署侧调研和后续闭环验证。
- 对部署侧已有参数做离线或真机 sweep：
  - `infer_interval`
  - `fusion_window`
  - `smooth_sigma`
  - `actions_per_chunk`
  - `chunk_size_threshold`

汇报原则：

- 推理延迟加速单独统计 TTFA、sample_actions latency、queue underrun、reaction latency。
- 不把 denoising speedup 写成机器人动作执行 speedup。

### 阶段 G：DLC 正式训练与多模型评测

触发条件：

- L20 上 metadata-conditioned / DemoSpeedup / RTC 至少有一条线稳定跑通。
- open-loop 指标显示有加速趋势。
- 轨迹可视化没有明显异常。

计划：

- 将配置迁移到 DLC。
- 跑更长 LoRA 或全量 fine-tune。
- 正式评测多个 checkpoint：
  - seatbelt full model
  - LoRA baseline
  - metadata-conditioned LoRA
  - DemoSpeedup-data LoRA
  - RTC LoRA
  - speedup model
  - 后续 RECAP/value 模型

产出：

- 完整横向评测 Markdown。
- 每个测试集的 duration summary。
- 可视化轨迹对比。
- 给 mentor 的结论：是否值得上真机闭环。

## 5. 当前问题清单

### 问题 1：seatbelt 全量模型 L20 跑不起来

原因：

- 单卡 46GB 在 checkpoint restore 或 train state 初始化阶段已经 OOM。

处理：

- L20 只做 LoRA 和小规模验证。
- 全量训练/正式大模型评测上 DLC。

### 问题 2：speedup 模型配置与当前代码不兼容

原因：

- 配置使用了 `window_datasets`，当前本地 `TrainConfig` 不支持。

处理：

- 同步对应更新代码。
- 或补兼容字段和下游逻辑。

### 问题 3：当前 speedup 指标还是 open-loop 估计

原因：

- 测试集只有固定示教动作，不能直接代表真机闭环完成时间。

处理：

- open-loop 阶段先筛选模型。
- 通过后再做真机闭环验证。

### 问题 4：LoRA smoke 结果统计量太小

原因：

- 只训练 1 step。
- 只测 1 batch。

处理：

- 已通过 Baseline v1 解决：训练 500 step，固定 8 个测试 repo，每个 repo 测 10 个 batch。

### 问题 5：pi0.7 完整标签体系工作量大

原因：

- 严格 pi0.7 不只需要 speed，还涉及 quality、mistake、subgoal images 等 metadata。

处理：

- 当前先做 pi0.7-lite：只自动生成速度标签。
- 后续如有效，再考虑引入质量/失败/错误类型标签。
