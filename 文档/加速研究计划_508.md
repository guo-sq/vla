# 加速研究计划：DemoSpeedup 与 pi0.6 RECAP 路线

日期：2026-05-08  
项目：`/root/workspaces/wujie_gsq/vla`

## 1. 研究目标

当前目标不是简单让模型输出更短的动作序列，而是训练出“更快且仍能到达目标”的策略。前期 baseline 结果说明，普通 LoRA imitation training 在 500、1000、1500 steps 下都没有稳定学到真正的加速能力，部分 speedup factor 还会被“Pred 很早停止或塌缩”误判为加速。因此后续加速研究必须同时解决两个问题：

1. **数据如何加速**：把示教中可加速的低风险片段压缩，保留高精度片段，构造更快的训练目标。
2. **模型如何知道何时该快、何时该稳**：引入 entropy / value / advantage / positive indicator，避免模型把所有动作都粗暴提前收敛。

本计划学习两条路线：

- **DemoSpeedup**：用 proxy policy 的不确定性或熵来判断示教片段重要性，对 casualness stage 加速，对 precision stage 保守。
- **pi0.6 RECAP**：用 value function 和 advantage-conditioned policy，让模型偏向高价值、高成功率、低失误的行为，而不是只模仿平均示教。

最终目标是形成一条适合 seatbelt 任务的加速训练流水线：

```text
原始示教数据
  -> proxy policy / entropy 估计
  -> 阶段划分：precision / transition / casual / static
  -> 生成加速 index map 或加速数据集
  -> value / advantage / speed-quality return 标注
  -> 训练加速策略
  -> 用 target-completion speedup + reached rate + 轨迹图评估
```

## 2. 当前基础情况

### 2.1 当前 baseline 训练方式

当前主要实验使用的是原项目已有的 pi0 LoRA 微调路线，并没有修改训练目标。典型配置为：

- 配置文件：`openpi_modified/src/openpi/configs/cfg_baseline_seatbelt_lora_1500_507.py`
- 基础 checkpoint：`84999`
- LoRA 微调：冻结大部分基础模型参数，仅训练低秩适配参数
- 训练目标：普通行为克隆，预测与示教一致的 action chunk
- 当前没有启用：
  - DemoSpeedup 数据重采样
  - speed prompt conditioning
  - RECAP advantage indicator
  - value head
  - sample weight 或 frame-level 加速权重

所以目前 baseline 本质上回答的是：

> 在原始慢速示教上继续 imitation training，模型会不会自然变快？

目前结论是：基本不会稳定变快。

### 2.2 当前评估协议

固定评估参数：

- `num_batches=10`
- `batch_size=1`
- `sample_steps=4`
- `duration_dims=0-5,7-12`
- `duration_abs_tol=0.02`
- `duration_rel_tol=0.05`
- `duration_stable_steps=3`
- `lazy_load_eval=1`

主要指标：

- MSE / MAE：动作数值误差
- old speedup factor：Pred 相对自身最终状态的完成时间
- target-completion speedup：Pred 到达 GT 最终目标的完成时间
- reached rate：Pred 是否真的到达 GT final target
- final target distance：Pred 最终动作与 GT final target 的距离
- action-space 2D/3D trajectory：直观看 Pred 是否跟踪 GT、是否塌缩、是否提前停止

后续所有加速实验都应以 **target-completion speedup + reached rate** 为主，old speedup 只能作为辅助参考。

### 2.4 当前模型质量判断

已评估过的模型包括：

- baseline 500 steps
- baseline 1500 steps
- bs8 500 steps
- bs8 1000 steps
- bs8 1500 steps

整体观察：

1. **1500 steps 不稳定优于 500 steps**  
   增加训练步数没有稳定改善真实目标到达能力，说明瓶颈不只是训练不够久。

2. **bs8 有一定帮助，但不是根本解决**  
   batch size 增大后训练更平滑，部分任务 MSE 或 reached rate 变好，但没有形成可靠加速策略。

3. **take_off_move 的高 speedup 存在误判风险**  
   可视化显示 Pred 可能很早进入稳定状态，但不一定真正跟踪到 GT 目标。它是后续判断“假加速”的重点任务。

4. **当前基础模型仍偏弱**  
   轨迹图中存在跟踪不充分、路径偏离、提前收敛、右臂/左臂不一致等问题。直接在这个基础上追求更快，容易放大错误。

因此后续路线应先做“保守加速 + 质量门控”，而不是直接大比例压缩全部示教。

## 3. DemoSpeedup 路线

### 3.1 核心思想

DemoSpeedup 的核心不是平均地把整条轨迹加速，而是根据动作不确定性或 entropy 判断每个阶段是否可以加速：

- 高 entropy / 高不确定性：模型不确定、动作精度要求高，通常对应接触、插入、避障、纠偏等 precision stage，应保守。
- 低 entropy / 低不确定性：模型比较确定，通常是移动、等待、回程等 casual stage，可适度加速。
- 静止或重复段：如果动作变化很小，可以压缩或删除一部分。

它适合当前项目的原因：

- 我们已有 baseline/proxy policy，可用于多次采样估计 action variance。
- 当前评估产物已有 `test_all_preds.npy` / `test_all_gts.npy`，可先离线验证 entropy 与轨迹质量的关系。
- 项目已有 action chunk 训练结构，适合通过 index map 方式做时序重采样。

### 3.2 第一阶段：entropy 估计

新增脚本：

```text
openpi_modified/scripts/estimate_action_entropy.py
```

输入：

- `--ckpt_dir`
- `--config_name`
- `--dataset_root`
- `--repo_id`
- `--num_samples_per_state 8 或 16`
- `--sample_steps 4`
- `--duration_dims 0-5,7-12`
- `--out_dir`

输出：

```text
entropy_results/<repo>/
  action_entropy.npy
  action_variance.npy
  entropy_summary.json
  entropy_curve.png
```

计算建议：

1. 对同一 observation 多次 sample action。
2. 在 `duration_dims=0-5,7-12` 上计算方差。
3. 对 horizon 维度聚合，得到每个 frame 或 chunk 的 uncertainty score。
4. 使用滑动平均平滑 entropy 曲线。
5. 对每个 episode 内部做 percentile normalization，避免不同 repo 量纲差异过大。

需要比较的 proxy：

- 原始基础 checkpoint `84999`
- bs8-500
- bs8-1000

判断标准：

- entropy 高峰是否出现在插入、取出、避障、纠偏等直觉上困难的阶段。
- entropy 低谷是否对应长距离移动、等待、回家等阶段。
- 如果 proxy 的 entropy 和轨迹质量完全不相关，不能直接用它生成加速数据。

### 3.3 第二阶段：阶段划分

新增脚本：

```text
openpi_modified/scripts/segment_demo_by_entropy.py
```

每个 frame/chunk 标注：

- `precision`：高 entropy、高曲率、高接近目标误差变化、接触/插入相关阶段
- `transition`：中等 entropy，动作变化明显但不极端
- `casual`：低 entropy，动作方向稳定
- `static`：动作变化很小，可压缩等待或重复动作

初始规则建议：

```text
entropy >= p75                  -> precision
p45 <= entropy < p75            -> transition
entropy < p45 and motion > eps  -> casual
motion <= eps                   -> static
```

其中 motion 可用 action 差分估计：

```text
motion_t = mean(abs(action_t - action_{t-1}) over dims 0-5,7-12)
```

输出：

```text
demospeedup_segments/<repo>/
  stage_labels.npy
  stage_summary.json
  stage_curve.png
```

### 3.4 第三阶段：生成加速 index map

第一版不要直接改原始数据文件，优先生成 index map，降低破坏数据对齐的风险。

新增脚本：

```text
openpi_modified/scripts/build_demospeedup_index_map.py
```

加速倍率建议从保守版本开始：

| Stage | 初始倍率 | 说明 |
|---|---:|---|
| precision | 1.00x | 不加速，保留精细动作 |
| transition | 1.10x 到 1.25x | 轻微加速 |
| casual | 1.50x | 明显加速 |
| static | 2.00x 或压缩重复帧 | 删除部分等待 |

输出：

```text
demospeedup_index_maps/<repo>/
  speedup_index_map.npy
  speedup_index_map.json
  speedup_preview.png
```

index map 格式建议：

```json
{
  "source_repo": "...",
  "source_episode": 0,
  "source_length": 120,
  "target_length": 86,
  "global_speedup": 1.395,
  "stage_speedup": {
    "precision": 1.0,
    "transition": 1.2,
    "casual": 1.5,
    "static": 2.0
  }
}
```

关键约束：

- image、state、action、timestamp、episode index 必须同步重采样。
- 不能只压缩 action，不压缩 observation。
- 不能让 chunk 跨 episode 边界。
- 对 final target 附近强制保留足够帧，避免模型学到提前停止。

该配置明确区分了两条 RECAP 路线：

1. `enable_recap=True`：模型内部 learned optimality embedding。
2. `AddAdvantageToPrompt`：通过文本 prompt 注入 positive / negative indicator。

第一版 seatbelt 建议优先使用 **indicator prompt 注入路线**，因为它更容易调试，也更容易在评估时固定为 positive。

### 4.2 RECAP 在加速任务中的角色

DemoSpeedup 负责回答：

> 哪些示教片段可以加速？

RECAP 负责回答：

> 哪些轨迹或片段值得模型偏向学习？

对 seatbelt 加速来说，RECAP 不应只用 success/failure 二分类，而应定义 speed-quality return：

```text
return = success_proxy
         - lambda_time * normalized_completion_time
         - lambda_final * final_target_distance
         - lambda_path * path_deviation
         - lambda_jitter * action_jitter
```

其中：

- `success_proxy`：是否到达 GT final target，或 replay/仿真中的成功信号。
- `normalized_completion_time`：target-completion time，相对 GT 越短越好。
- `final_target_distance`：最终状态与目标距离。
- `path_deviation`：Pred 与 GT 主路径偏离程度。
- `action_jitter`：动作抖动，避免速度提升靠不稳定输出。

### 4.3 第一版 offline RECAP

在没有真实 rollout reward 或人工 intervention 数据前，先使用已有评估数据构造 pseudo advantage。

新增脚本：

```text
openpi_modified/scripts/build_speed_quality_returns.py
```

输入：

- `duration_summary.json`
- `duration_per_chunk.csv`
- `test_all_preds.npy`
- `test_all_gts.npy`

输出：

```text
recap_speed_quality/<repo>/
  speed_quality_returns.csv
  advantage_labels.csv
  advantage_summary.json
```

positive 样本建议：

- reached target
- target-completion speedup > 1.05 或 completion time 明显短于 GT
- final target distance 低
- MSE / path deviation 不超过当前 baseline 的合理阈值

negative 样本建议：

- old speedup 高但 target not reached
- final target distance 大
- 轨迹明显提前塌缩
- action jitter 高
- collision recovery / avoid collision 中偏离严重

训练方式：

- 在 prompt 中注入：

```text
Advantage: positive
```

或：

```text
Advantage: negative
```

- 训练时保留一定 dropout，避免模型完全依赖 indicator。
- 推理时固定使用 positive indicator。

### 4.4 第二版 value model RECAP

在 offline pseudo return 可用后，再走更接近 pi0.6 的 value pipeline：

```text
train value model
  -> compute_values.py
  -> compute_advantages.py
  -> AddAdvantageToPrompt
  -> train advantage-conditioned policy
```

需要新增 seatbelt 专用 value config：

```text
openpi_modified/src/openpi/configs/cfg_pi06_train_value_seatbelt_speed_quality_508.py
openpi_modified/src/openpi/configs/cfg_pi06_recap_indicator_seatbelt_speed_quality_508.py
```

value target 不应只看 episode 成败，而应包含速度质量：

```text
V(o_t, a_t) ~= expected speed-quality return from current state
```

如果后续接入真实部署或仿真 rollout，则可以进一步加入：

- 人工 intervention
- collision / near collision
- task success
- elapsed time
- recovery cost

### 4.5 RECAP 验收标准

RECAP 路线不一定第一版就显著提高 speedup，但至少应满足：

- positive prompt 下 reached rate 高于无 prompt baseline。
- false-fast 样本减少。
- take_off_move 不再出现明显提前塌缩。
- collision / recovery 类任务 final target distance 不恶化。
- trajectory 图中 positive 输出更接近 GT 主路径，negative 输出允许较差，但不参与最终部署。

## 5. DemoSpeedup + RECAP 组合策略

两者结合时，建议采用“加速由 entropy 决定，质量由 advantage 过滤”的原则。

### 5.1 组合规则

只在以下条件同时满足时明显加速：

```text
stage in {casual, static}
and entropy is low
and advantage is positive or unknown-but-safe
and near-final-target is false
```

以下情况不加速或轻微加速：

```text
stage == precision
or entropy is high
or advantage is negative
or final target distance risk is high
or task is collision recovery / avoidance critical segment
```

### 5.2 训练数据混合

建议构造三类数据：

1. 原始示教数据：保证基础跟踪能力。
2. DemoSpeedup 加速示教：提供更短时间目标。
3. RECAP positive 数据：告诉模型哪些行为更值得模仿。

初始混合比例：

```text
original : demospeedup : recap_positive = 0.4 : 0.4 : 0.2
```

如果模型开始偏离或不收敛，提高 original 比例。  
如果模型稳定但不够快，提高 demospeedup 比例。

### 5.3 推理方式

第一版推理固定：

```text
Advantage: positive
```

暂不引入复杂的运行时 speed level 控制。等基础效果稳定后，再考虑加入：

- `Speed: normal`
- `Speed: fast`
- `Speed: conservative`

## 8. 风险与应对

### 风险 1：proxy policy 本身太差，entropy 不可信

应对：

- 比较基础 checkpoint、bs8-500、bs8-1000 三个 proxy。
- 不直接相信 entropy，叠加 action curvature 和 final target distance 风险。
- 第一版只加速低 entropy 且低 motion-risk 的片段。

### 风险 2：speedup 指标再次误判

应对：

- old speedup 不作为主指标。
- 强制报告 target-completion speedup 和 reached rate。
- 对高 speedup 但 target not reached 的 chunk 标记为 false-fast。

### 风险 3：加速数据破坏 image/state/action 对齐

应对：

- 第一版用 index map，不直接改原始数据。
- 写 `validate_speedup_index_map.py` 检查 episode 边界、长度、chunk 对齐。
- 每个 repo 抽样可视化原始轨迹与加速后轨迹。

### 风险 4：RECAP label 来自 open-loop 评估，和真实部署成功不一致

应对：

- offline RECAP 只作为第一版。
- 后续必须接入仿真或真实 rollout reward。
- intervention/self-play 数据进入后，替换 pseudo return。

### 风险 5：模型学到“快但粗糙”

应对：

- reward 中加入 final target distance、path deviation、jitter。
- precision stage 不加速。
- original data 与 accelerated data 混合训练。

## 10. 推荐的下一步

优先做 DemoSpeedup 的最小闭环：

```text
entropy 估计
  -> stage label
  -> index map
  -> take_off 单任务训练
  -> target-completion speedup 复评
```

原因是当前最大问题是模型没有真实学到“加速后的示教目标”。RECAP 很重要，但它更像质量选择和偏好强化；如果没有先构造合理的快速示教，RECAP 只能告诉模型“哪些慢动作比较好”，不一定能产生真正加速。

因此建议顺序为：

1. DemoSpeedup v0 先跑通。
2. 用新的 target-completion 指标确认没有假加速。
3. 再引入 RECAP positive indicator。
4. 最后做 DemoSpeedup + RECAP 组合。
