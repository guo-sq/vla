## 系统级代理开关

本方案提供系统级代理环境变量开关，但默认没有启用。

开启系统级代理：

```bash
cd /root/workspaces/wujie_gsq/tools/mihomo
scripts/enable_system_proxy.sh
```

这会创建：

```bash
/etc/profile.d/wujie_gsq_mihomo_proxy.sh
```

影响范围：

- 新开的 login shell 会自动带上代理环境变量
- 支持 `HTTP_PROXY`、`HTTPS_PROXY`、`ALL_PROXY` 的程序会走代理
- 不会自动接管所有网络流量
- 不会启用 TUN

关闭系统级代理：

```bash
cd /root/workspaces/wujie_gsq/tools/mihomo
scripts/disable_system_proxy.sh
```

如果旧终端里已经加载过代理变量，需要手动清掉：

```bash
unset http_proxy https_proxy HTTP_PROXY HTTPS_PROXY ALL_PROXY all_proxy
```

## TUN 状态

配置文件里保留了 TUN 配置，但默认关闭：

不建议在共享 `root` 服务器上默认启用 TUN，因为 TUN 会进入系统网络层，可能影响整台服务器的网络行为。

如果要完全回退本次 Mihomo/Clash 安装，按下面顺序执行。

第一步，进入目录：

# 2026-04-27 后续实际保留改动：pproxy HTTP -> SOCKS5 方案

## 背景

Mihomo/Clash 方案已经回退。当前服务器上实际保留的是更轻量的 `pproxy` 方案，用来解决 VS Code Remote / Codex / Claude Code 这类工具无法直接使用 SOCKS5 代理的问题。

核心链路：

```text
VS Code Remote / Codex / Claude Code
  -> HTTP proxy: http://127.0.0.1:7897
  -> pproxy
  -> SOCKS5 remote proxy
  -> 外网
```

注意：这里的 `127.0.0.1:7897` 是 DSW 服务器自己的本地端口，不是本地电脑端口。

## 为什么这样做

DSW 服务器可以访问公网，但默认出口访问某些服务时会遇到地区限制，例如 Codex 登录时曾出现：

```text
403 Forbidden: Country, region, or territory not supported
```

VS Code Remote 和部分 Node.js 工具更容易识别 HTTP 代理环境变量，不一定能直接使用 SOCKS5。因此用 `pproxy` 在服务器本地启动一个 HTTP 代理入口，再把流量转发到远端 SOCKS5 代理。

## 已安装内容

服务器上已安装：

```bash
python3-pip
pysocks
pproxy
```

其中 `python3-pip` 是通过 `apt-get install -y python3-pip` 安装的；`pysocks` 和 `pproxy` 是通过 `python3 -m pip install pysocks pproxy` 安装的。

## 已修改文件

### 1. `/root/.proxy_control`

新增了代理开关函数，当前文件位置：

```bash
/root/.proxy_control
```

该文件里保存了以下命令：

```bash
proxy_on
proxy_off
proxy_stop
proxy_status
```

代理账号凭据保存在此文件中，属于敏感信息。文档中不记录完整凭据。

### 2. `/root/.bashrc`

追加了这一行：

```bash
[ -f /root/.proxy_control ] && source /root/.proxy_control
```

作用：新开 Bash 终端时，只加载代理辅助函数，但默认不自动开启代理。

### 3. `/root/.zshrc`

追加了这一行：

```bash
[ -f /root/.proxy_control ] && source /root/.proxy_control
```

作用：新开 Zsh 终端时，只加载代理辅助函数，但默认不自动开启代理。

### 4. `/root/.vscode-server/data/Machine/settings.json`

新增或更新了 VS Code Remote 设置：

```json
{
    "remote.downloadExtensionsLocally": true,
    "remote.SSH.httpProxy": "",
    "http.proxy": "http://127.0.0.1:7897",
    "http.proxyStrictSSL": false,
    "http.proxyAuthorization": null,
    "claudeCode.environmentVariables": [
        { "name": "HTTP_PROXY", "value": "http://127.0.0.1:7897" },
        { "name": "HTTPS_PROXY", "value": "http://127.0.0.1:7897" },
        { "name": "http_proxy", "value": "http://127.0.0.1:7897" },
        { "name": "https_proxy", "value": "http://127.0.0.1:7897" }
    ]
}
```

作用：让 VS Code Remote 和 Claude Code 扩展知道 HTTP 代理入口是 `http://127.0.0.1:7897`。

## 当前开关行为

当前方案不是全服务器透明代理，也不是 TUN。它只影响显式读取代理环境变量或 VS Code Remote 设置的程序。

### 查看状态

```bash
proxy_status
```

输出含义：

```text
http_proxy=<unset>
pproxy: not running
```

表示当前 shell 没有代理变量，`pproxy` 也没运行。

```text
http_proxy=http://127.0.0.1:7897
pproxy: running on 127.0.0.1:7897
```

表示当前 shell 会走代理，且 `pproxy` 正在监听 `7897`。

### 开启代理

```bash
proxy_on
```

它会做两件事：

- 如果 `pproxy` 没跑，就启动 `pproxy` 监听 `http://127.0.0.1:7897`
- 给当前 shell 设置 `http_proxy`、`https_proxy`、`HTTP_PROXY`、`HTTPS_PROXY`

### 关闭当前 shell 代理环境变量

```bash
proxy_off
```

它只清除当前 shell 的代理环境变量，不杀掉 `pproxy`。这样 VS Code Remote 仍然可以继续使用 `127.0.0.1:7897`。

### 停止 pproxy

```bash
proxy_stop
```

它会停止 `pproxy` 进程。执行后，VS Code Remote 里配置的 `http://127.0.0.1:7897` 也会失效，直到再次执行 `proxy_on`。

## 验证方式

手动指定代理验证：

```bash
curl -I -x http://127.0.0.1:7897 https://api.anthropic.com --max-time 15
```

如果看到类似：

```text
HTTP/1.1 200 Connection established
HTTP/2 404
```

说明代理链路已经打通。`404` 在这里不是网络失败，它表示请求已经到达目标服务，只是访问路径不是业务接口。

## 和 GitLab / GitHub 的关系

本次代理能解决外网访问问题，例如 GitHub、Anthropic、OpenAI 等公网服务。

它不能解决公司内网域名解析问题，例如：

```text
gitlab.anyverse.work
```

服务器上曾验证：

```text
fatal: unable to access 'https://gitlab.anyverse.work/...': Could not resolve host: gitlab.anyverse.work
```

原因是 DSW 服务器没有公司内网 DNS / VPN / GitLab 访问通路。外网代理不会自动让服务器获得公司内网域名解析能力。

因此当前代码同步路线改为：

```text
本地内部 GitLab 仓库
  -> 本地推送到 GitHub: guo-sq/vla
  -> DSW 服务器从 GitHub 拉取
```

## 影响范围

当前方案的影响范围：

- `proxy_on` 只影响当前 shell 及其后续子进程
- `proxy_off` 只清除当前 shell 的代理变量
- `proxy_stop` 会停止 `pproxy`，会影响依赖 `127.0.0.1:7897` 的 VS Code Remote / 插件
- `/root/.bashrc` 和 `/root/.zshrc` 只加载函数，默认不自动开启代理
- `/root/.vscode-server/data/Machine/settings.json` 会让 root 用户下的 VS Code Remote 使用 `127.0.0.1:7897`

当前方案没有做：

- 没有修改 `/etc/profile.d`
- 没有启用 TUN
- 没有修改系统路由
- 没有接管整台服务器所有网络流量

## 回滚 pproxy 方案

如需完全回滚当前 `pproxy` 方案，执行：

```bash
proxy_stop
```

删除辅助文件：

```bash
rm -f /root/.proxy_control
```

从 `/root/.bashrc` 和 `/root/.zshrc` 中删除：

```bash
[ -f /root/.proxy_control ] && source /root/.proxy_control
```

删除或手动编辑 VS Code Remote 设置：

```bash
rm -f /root/.vscode-server/data/Machine/settings.json
```

如果只想去掉代理字段，不要删除整个 VS Code 设置文件，应手动移除：

```json

```

如需删除 GitHub deploy key：

```bash
rm -f /root/.ssh/id_ed25519_guosq_vla /root/.ssh/id_ed25519_guosq_vla.pub
```

并从 `/root/.ssh/config` 删除：

```ssh

  HostName github.com
  User git
  IdentityFile ~/.ssh/id_ed25519_guosq_vla
  IdentitiesOnly yes
```

最后也需要到 GitHub 仓库 `guo-sq/vla` 的 Deploy keys 页面删除对应 key。
